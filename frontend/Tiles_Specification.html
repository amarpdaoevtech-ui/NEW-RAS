Overview — What is a Tile?

The dashboard is made up entirely of tiles — self-contained React components that each display one category of vehicle data. A tile does not know where it lives on screen; it simply receives data as props and renders itself. The layout engine (TilesEngine.jsx) is responsible for reading the JSON configuration, positioning each tile on a CSS grid, and passing the correct data from the Zustand store into each tile's props.

There are six tile types in total. Each type has a specific visual form — a graph, a ring gauge, a bar, a status indicator, a large number, or a mode selector. By combining these six types across 11 grid positions, the dashboard can show every relevant metric the vehicle produces without any custom one-off components.

Configuration-driven: The entire layout — which tile goes where, how large it is, and which data field it reads — is declared in tilesConfig.json. No JSX changes are needed to add, remove, or reposition tiles.

How a tile gets its data

  1. The backend broadcasts a bms_update WebSocket event (1–10 times per second).
  2. The Zustand store receives the payload and updates its telemetry object.
  3. Each tile component subscribes to the specific store field(s) it needs — nothing more.
  4. React re-renders only the tiles whose subscribed data has changed (max 20 FPS throttle).

Grid System & Sizing Reference

The canvas is fixed at 1024 × 600 px to match the 7-inch HDMI LCD exactly. A 4-column, 6-row CSS grid divides this canvas into cells. Tiles occupy one or more cells. All pixel sizes are deterministic — no fluid or percentage-based sizing.

ParameterValueNotes
Canvas width1024 pxMatches LCD horizontal resolution
Canvas height600 pxMatches LCD vertical resolution
Columns4Each column = 246 px
Rows6Each row = 91 px
Gap between cells8 pxApplied horizontally and vertically
Padding X (left/right)8 pxOuter canvas padding
Padding Y (top/bottom)7 pxOuter canvas padding

Tile pixel size formula

width  = (w × colWidth)  + ((w - 1) × gap)   →  e.g. 2 col = (2×246) + (1×8) = 500 px
height = (h × rowHeight) + ((h - 1) × gap)   →  e.g. 3 row = (3×91)  + (2×8) = 289 px

TileType1 — Live Line Graph
Type 1 Live Line Graph TileType1.jsx  ·  tile_1

This is the largest tile on the dashboard and the most data-rich. It draws a scrolling time-series line graph on an HTML Canvas element, showing the last two hours of vehicle data across up to four channels simultaneously — speed, voltage, current, and temperature. The rider can see trends at a glance: is speed climbing steadily, did voltage drop suddenly, is temperature creeping up? A dashed grid and translucent area fill make the chart readable even in bright outdoor conditions.

TileType2 — Circular Ring Gauge
Type 2 Circular Ring Gauge TileType2.jsx  ·  tile_2, tile_8

The ring gauge is purpose-built for percentage values — most critically State of Charge (SOC) and State of Health (SOH). A 270° SVG arc sweeps clockwise from the bottom-left to the bottom-right of the circle, filling proportionally to the value. The percentage is displayed in large text at the centre of the ring. The colour of the arc moves from green (high) through blue (normal) to red (critically low), and a pulsing glow animation activates when the value enters a danger zone — giving the rider an immediate warning without taking their eyes off the road for more than a glance.

TileType3 — Progress Bar
Type 3 Progress Bar TileType3.jsx  ·  tile_3, tile_7

The progress bar shows how much of a defined range is currently being used — most commonly throttle position (0–100%) and power output (0–3,160 W). The bar fills horizontally from left to right, and its colour shifts dynamically as the value climbs: green at low levels signals efficient riding, amber in the mid-range indicates increasing load, and red at the top warns the rider they are demanding peak power from the motor. This colour gradient gives instant, eyes-up feedback with zero cognitive load.

TileType4 — Status Display
Type 4 Status Display TileType4.jsx  ·  tile_4, tile_10, tile_11

The status display tile communicates binary system states — something is either active or it is not. A pulsing coloured dot (green when active, grey when inactive, red when faulted) does the primary work; a large number below it provides a secondary metric in context. Three tiles on the current layout use this type: BMS connection status, brake and cruise control state, and ESP32 heartbeat alongside the Pi's battery percentage. This tile is the system's health panel — if something goes wrong, these three tiles will show it first.

TileType5 — Hero Number
Type 5 Hero Number TileType5.jsx  ·  tile_5, tile_9

Some numbers are important enough to stand alone. The hero number tile renders a single value as large as the tile allows — 64 px bold — with a small label above and a unit below. There is no scale, no bar, no gauge. The number simply speaks for itself. It is used for the Distance to Empty (DTE) and the lifetime odometer. A count-up animation runs whenever the value changes, using the useCountUp hook — this adds a subtle analogue feel and prevents jarring number jumps, which is particularly important for the odometer which increments frequently during a ride.

TileType6 — Drive Mode Selector
Type 6 Drive Mode Selector TileType6.jsx  ·  tile_6

The drive mode tile shows the currently active riding mode as reported by the ESP32 co-processor — one of three options: ECO, NOR (Normal), or HIGH. All three modes are always visible as badges; the active one is filled with its characteristic colour and the other two are shown as dim outlines. Mode switching is controlled by a physical button on the vehicle, not by the touchscreen — this tile is read-only. The active mode also feeds the DTE calculation engine, which uses different efficiency coefficients for each mode, so what the rider sees here directly influences the range estimate shown in tile_5.

Alert & Threshold System

Every tile type that shows a bounded value has built-in threshold awareness. When a value enters a warning or danger zone, the tile activates a visual alert — a colour change, a pulsing glow, or both. Alerts are visual only (no audio in v2). All thresholds are configurable per tile instance via props, so the same component can behave differently depending on what it is measuring.

Animation specification

All alert animations are injected as a <style> tag scoped to each tile instance. This avoids global CSS pollution when multiple tiles of the same type appear simultaneously on screen. 

/* Danger pulse — 1.5 s, red glow */
@keyframes tile-danger-pulse {
  0%,100% { box-shadow: 0 0  0px 0px rgba(239,68,68,0.0); }
  50%     { box-shadow: 0 0 16px 4px rgba(239,68,68,0.6); }
}

/* Warning pulse — 2 s, amber glow */
@keyframes tile-warn-pulse {
  0%,100% { box-shadow: 0 0  0px 0px rgba(245,158,11,0.0); }
  50%     { box-shadow: 0 0 12px 3px rgba(245,158,11,0.5); }
}

/* Error dot fast pulse — 0.8 s */
@keyframes dot-error-pulse {
  0%,100% { opacity: 1;   }
  50%     { opacity: 0.3; }
}
